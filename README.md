### Выбор UI-сценария

В рамках сравнения был выбран типовой пользовательский сценарий работы с задачами. Сначала пользователю нужно увидеть список всех задач, где отображаются только идентификатор, заголовок и статус выполнения. Затем, выбрав конкретную задачу, он хочет посмотреть её полную информацию: описание, дату выполнения, даты создания и последнего обновления. В системе предусмотрена возможность создания новой задачи, для чего необходимо указать заголовок, описание и дату выполнения. Таким образом, сценарий включает три основных действия: просмотр сокращённого списка, просмотр деталей одной задачи и создание новой задачи.

## Сравнительный анализ

### Количество запросов

| Критерий | REST | GraphQL |
|----------|------|---------|
| Экран списка | 1 запрос | 1 запрос |
| Карточка детали | 1 запрос | 1 запрос |
| Создание задачи | 1 запрос | 1 запрос |
| **Итого** | **3 запроса** | **3 запроса** |


### Объём данных (сравнение размеров ответов)

| Запрос | REST (байт) | GraphQL (байт) | Разница |
|--------|-------------|----------------|---------|
| Список задач (2 задачи) | ~850 байт | ~350 байт | GraphQL на ~40% меньше |
| Детали задачи | ~650 байт | ~350 байт | GraphQL почти вдвое меньше |
| Создание задачи | ~650 байт | ~350 байт | GraphQL почти вдвое меньше |


### Ошибки и статусы

| Аспект | REST | GraphQL |
|--------|------|---------|
| HTTP статусы | Разные статусы для разных ошибок (200, 400, 401, 404, 500) | Всегда 200 (даже при ошибках) |
| Формат ошибки | `{"error": "сообщение"}` в теле ответа | `{"errors": [{"message": "..."}]}` в теле |
| Успешный ответ | Данные в теле + статус 200 | Данные в поле `data` + статус 200 |
| Частичный успех | Не поддерживается | Можно вернуть и данные, и ошибки |


### Кэширование

| Аспект | REST | GraphQL |
|--------|------|---------|
| Кэширование по URL | Просто — каждый URL уникален | Сложно — все запросы на один URL |
| HTTP кэши | Работает стандартно | Не работает из-за POST запросов |
| Кэширование на уровне данных | Требует доп. решений | Требует доп. решений |
| Persisted Queries | Не применимо | Помогает с кэшированием |


### Вывод

По количеству запросов REST и GraphQL оказались одинаковыми — в обоих случаях нужно сделать три запроса, чтобы получить список задач, открыть детали и создать новую.
А вот по объёму данных GraphQL выигрывает заметно. При запросе списока задач через REST, сервер возвращал все поля — и описание, и даты, и всё остальное, хотя нужны были только заголовок и статус. Из-за этого ответ весил около 850 байт. GraphQL же вернул ровно то, что запрошено, и ответ получился почти в два с половиной раза меньше — всего 350 байт. То же самое с деталями задачи и с созданием: GraphQL всегда отдаёт только запрошенные поля, а REST — всё подряд. 
По обработке ошибок REST выигрывает, потому что разные ошибки имеют разные HTTP статусы. Когда 401 — сразу понятно, что проблема с авторизацией, 404 — задача не найдена, 500 — что то сломалось на сервере. В GraphQL же все ответы приходят со статусом 200, и чтобы понять, что пошло не так, нужно каждый раз заглядывать в тело ответа в поле errors. Это неудобно для мониторинга и отладки.
С кэшированием REST тоже проще. Каждый эндпоинт имеет свой уникальный URL, и браузер или CDN могут легко закэшировать ответы. GraphQL же все запросы отправляет на один и тот же адрес методом POST, поэтому стандартное HTTP-кэширование не работает.
Если проект простой, данные меняются нечасто и важна простота кэширования — лучше взять REST. А если нужно экономить трафик, данные сложные и разные клиенты запрашивают разные поля — тогда GraphQL будет удобнее. 


### Ответы на контрольные вопросы

1. Что такое over-fetching и under-fetching?
Over-fetching — это когда сервер возвращает больше данных, чем нужно клиенту. Например, в REST я запрашиваю список задач, а получаю ещё и описание, и даты, которые на экране списка не нужны. Under-fetching — это когда одного запроса недостаточно, и клиенту приходится делать несколько запросов подряд, чтобы получить все нужные данные.

2. Почему GraphQL может быть опасен без ограничений сложности запросов?
Клиент может отправить очень сложный вложенный запрос, который заставит сервер выполнить множество операций с базой данных одновременно. Например, запросить не только задачи, но и для каждой задачи ещё и историю изменений, и комментарии. Без ограничений злоумышленник может таким запросом перегрузить сервер и вызвать отказ в обслуживании (DoS-атаку). Поэтому на реальных проектах всегда ставят лимиты на глубину запросов и количество возвращаемых записей.

3. Почему кэширование REST обычно проще?
В REST каждый эндпоинт имеет свой уникальный URL, и ответ зависит только от этого URL и HTTP-метода. Браузеры, CDN и прокси-серверы умеют кэшировать GET-запросы по URL «из коробки». В GraphQL все запросы отправляются на один и тот же URL методом POST, а тело запроса (сам GraphQL-запрос) может быть разным. Поэтому стандартное HTTP-кэширование не работает, и приходится придумывать обходные пути — persisted queries, кэширование на уровне данных или использовать специальные клиенты вроде Apollo Client.

4. Как отличаются ошибки REST и GraphQL с точки зрения клиента?
В REST клиент может понять, что пошло не так, по HTTP-статусу: 401 — не авторизован, 404 — не найдено, 500 — ошибка сервера. Это удобно для обработки и мониторинга. В GraphQL почти все ответы приходят со статусом 200 OK, даже если внутри ошибка. Чтобы узнать, что случилось, клиенту нужно каждый раз проверять наличие поля errors в теле ответа. Это менее удобно, зато позволяет вернуть частичный успех — например, и данные, и ошибку одновременно.

5. В каких случаях вы бы выбрали REST, а в каких GraphQL?
REST я бы выбрала, когда нужно простое кэширование, API публичное и стабильное, и клиенты всегда запрашивают одни и те же данные. Например, новостной сайт или API для погоды. GraphQL я бы выбрала, когда у приложения много разных клиентов (веб, мобильное приложение, внешние интеграции), и каждому нужны разные наборы полей, или когда нужно избежать нескольких последовательных запросов к серверу. Например, сложные дашборды, социальные сети, агрегаторы данных.
